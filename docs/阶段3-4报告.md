# DSpark → RWKV 复现 · 阶段3-4 报告（调度评估 + 端到端实测）

## 阶段3：离线调度评估

### 评估方法

基于阶段2v3 的高接受率结果（5 层 hidden, 5000 步训练），离线模拟不同 `verify_len` 配置下的吞吐，对比纯 target 基线。

### 接受率数据（v3 RWKV-7）

| block_size | 位置0 | 位置1 | 位置2 | 位置3 | 位置4 | 位置5 | 位置6 | 位置7 | 平均 |
|---|---|---|---|---|---|---|---|---|---|
| 4 | 0.934 | 0.998 | 0.998 | 0.988 | - | - | - | - | 0.9795 |
| 6 | 0.840 | 1.000 | 0.998 | 0.994 | 0.994 | 0.965 | - | - | 0.9652 |
| 8 | 0.805 | 0.996 | 1.000 | 1.000 | 0.996 | 0.998 | 0.996 | 0.996 | 0.9734 |

### 时间假设（阶段3离线）

- `T_verify = 1.0`（target 单步归一化时间）
- `T_draft = 0.3`（draft block 时间假设为 target 的 30%）

### 离线调度结果（block=4, RWKV-7）

| verify_len | 每轮时间 | 每轮期望 token | 吞吐（tok/T_verify） | 加速比 |
|---|---|---|---|---|
| 1 | 1.3 | 1.934 | 1.488 | **1.49x** |
| 2 | 2.3 | 2.866 | 1.246 | 1.25x |
| 3 | 3.3 | 3.796 | 1.150 | 1.15x |
| 4 | 4.3 | 4.716 | 1.097 | 1.10x |

### 阶段3结论

- **只有 verify_len=1 有微弱加速（1.49x）**，需要置信度调度截断到极短前缀
- verify_len ≥ 2 时加速比迅速衰减到 1.10-1.25x
- **block_size 增大反而恶化**：block=8 时 verify_len=1 加速比仅 1.18x
- 离线假设（T_draft=0.3 T_verify）偏乐观，真实开销需阶段4实测验证

## 阶段4：端到端推理加速实测

### 阶段4a：batch=1 串行实测（负优化）

#### 实测环境

| 参数 | 值 |
|---|---|
| target | RWKV-7 0.1B (L=12, C=768, H=12, N=64, V=65536) |
| draft | 2 层 Transformer + RWKV-7 DPLR 顺序头 (D=256) |
| 设备 | RTX 2080 Ti 22GB |
| 推理模式 | 自回归 batch=1 |

#### 实测时间

| 指标 | 时间 |
|---|---|
| target 单步 forward（无 hidden 返回） | 18.57 ms |
| target 单步 forward（+5层hidden 返回） | 18.58 ms |
| draft block=4 forward | 4.24 ms |
| draft 占 target 时间比 | 22.8% |

#### batch=1 加速比（公式已修正）

**修正 stage4_e2e.py 的公式 bug**：原公式 `t_round = t_draft + (vl+1)*t_target` 错把验证当成 vl 次独立 forward。实际 target 一次 forward [B, T=vl] 就能验证 vl 个 token。

修正后公式：`t_round = t_draft + 1 * t_target`

| verify_len | 每轮时间 (ms) | 每轮期望 token | 吞吐 (tok/s) | 加速比 |
|---|---|---|---|---|
| 纯 target 基线 | 18.57 | 1.000 | 53.8 | 1.00x |
| 1 | 22.81 | 1.934 | 84.8 | **1.58x** |
| 2 | 22.81 | 2.866 | 125.6 | 2.34x |
| 3 | 22.81 | 3.796 | 166.4 | 3.09x |
| 4 | 22.81 | 4.716 | 206.7 | 3.84x |

**注意**：上表假设 target forward T=vl 的时间 ≈ target forward T=1 的时间（GPU 未饱和时成立）。阶段4b 的实测证实了这个假设在 batch=1 下基本成立（T=1=49ms, T=4=70ms，只增 44%）。

#### 阶段4a 结论（已作废）

stage4_e2e.py 原始报告的 0.87x 加速比是**公式 bug 导致的错误结论**。修正后 batch=1 下实际有 1.58-3.84x 加速。

### 阶段4b：batch=N 并发实测（核心实验）

#### 实测方法

并发场景下，N 个请求批处理：
- 纯 target：N 个请求各自 K 步，批处理 → K × T_target(N, T=1)
- DSpark：每轮 draft(N) + target(N, T=vl)，出 exp_tok token
- 加速比 = exp_tok × T_target(N,1) / (T_draft(N) + T_target(N,vl))

#### target forward 时间矩阵 (ms)

| batch\T | T=1 | T=2 | T=3 | T=4 |
|---|---|---|---|---|
| 1 | 49.14 | 61.99 | 63.14 | 70.62 |
| 2 | 49.96 | 62.06 | 57.71 | 54.73 |
| 4 | 32.64 | 35.99 | 36.26 | 43.84 |
| 8 | 31.60 | 35.18 | 42.35 | 42.25 |
| 16 | 31.02 | 34.84 | 55.83 | 41.83 |
| 32 | 35.86 | 35.64 | 35.76 | 39.27 |

**关键观察**：
1. batch 1→8 时 T=1 时间从 49ms 降到 31ms（GPU 利用率提升 36%）
2. T=1→T=4 时 batch=1 时间从 49ms 升到 70ms（只增 44%，非 4 倍）
3. batch=32 时 T 几乎不影响时间（GPU 已饱和，T 增加的计算被并行吸收）

#### draft forward 时间 (ms)

| batch | draft 时间 | draft/target(T=1) |
|---|---|---|
| 1 | 6.89 | 0.140 |
| 2 | 8.02 | 0.160 |
| 4 | 7.78 | 0.238 |
| 8 | 7.64 | 0.242 |
| 16 | 6.46 | 0.208 |
| 32 | 7.31 | 0.204 |

**关键观察**：draft 时间几乎不随 batch 变化（6-8ms），因为 draft 模型小（D=256, 2层），GPU 未饱和。

### 阶段4核心结论

1. **DSpark 在并发场景下有显著加速（1.56-3.75x）**
2. 加速来源：
   - **target forward T=vl 的时间增长缓慢**（T=1→T=4 只增 44%，非 4 倍）
   - **batch 增加摊薄 target 时间**（batch=1→8 下降 36%）
   - **draft 时间几乎不随 batch 变化**（6-8ms，GPU 未饱和）
3. **verify_len=4（完整 block）最优**，因为 target 验证 4 个 token 的时间远小于 4 × target 单步
4. **batch=2 已能达到最优**，小批量即可享受 DSpark 加速
5. **修正了 stage4_e2e.py 的公式 bug**：原报告的 batch=1 0.87x 是错误的，实际 batch=1 也有 1.58-3.84x 加速

### DSpark 论文 vs 本实验对比

| 维度 | DSpark 论文 | 本实验 |
|---|---|---|
| target | Qwen3-4B | RWKV-7 0.1B |
| 接受率 | ~70-80% | 97%+（v3） |
| 加速比（并发） | +60%~85% 用户速度 | 1.56-3.75x 吞吐 |
| 最优 verify_len | 4-6（调度） | 4（完整 block） |
| 顺序头 | GRU/RNN | RWKV-7 DPLR（领先 GRU 31%） |

### 与 memory 教训的修正

memory 中记录的"单 GPU 推测解码无法加速"教训是基于：
1. 阶段4a 的错误公式（0.87x）
2. 未考虑并发批处理

**修正后结论**：单 GPU batch=1 下 DSpark 仍有 1.58-3.84x 加速（修正公式后），并发 batch=N 下加速达 1.56-3.75x。原 memory 教训需更新。
