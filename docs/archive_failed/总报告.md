# DSpark → RWKV-7 复现 · 总报告

## 项目目标

复现 DSpark 论文（DeepSeek + 北大，2025）的半自归草稿架构 + 置信度调度验证，验证在 RWKV-7 target 上的可行性。

- 论文：https://github.com/deepseek-ai/DeepSpec/blob/main/DSpark_paper.pdf
- 推理库参考：https://github.com/BlinkDL/Albatross
- RWKV-7 公式参考：https://github.com/BlinkDL/RWKV-LM/blob/main/RWKV-v7/rwkv_v7_numpy.py

## 四阶段路线

| 阶段 | 内容 | 状态 | 结果 |
|---|---|---|---|
| 1 | 顺序头单测（合成任务） | ✅ | RWKV-7 DPLR 可用，token shift + LayerNorm 是必要条件 |
| 2 | 真实 RWKV target + DSpark 双轨 | ✅ | 接受率 97%+，RWKV 领先 GRU 31% |
| 3 | 置信度头 + 离线调度评估 | ✅ | 离线预测 verify_len=1 加速 1.49x |
| 4 | 端到端推理加速实测 | ✅ | 并发场景 1.56-3.75x 加速 |

## 阶段1：顺序头单测（合成任务）

### 实验

在合成任务（单 key / 双 key）上验证 RWKV-7 DPLR 作为 DSpark 顺序头的可行性，并进行 token shift + LayerNorm 的消融实验。

### 关键结果

| 头 | 单 key 平均 | 双 key 平均 |
|---|---|---|
| None (baseline) | 0.124 | 0.109 |
| GRU (DSpark) | 0.838 | 0.797 |
| RWKV baseline（无 shift 无 LN） | 0.842 | 0.805 |
| 实验1（+shift） | 0.912 | 0.887 |
| 实验1+2（shift+LN） | **0.954** | **0.929** |

### 消融结论

- **token shift 是必要条件**：+9% 接受率
- **LayerNorm 是必要条件**（RWKV-7 推荐 LN 不是 RMSNorm）：+4% 接受率
- 两者结合在双 key 任务上提升 +12.4%

详见：`参考/dspark_rwkv/阶段1实验报告.md`、`参考/dspark_rwkv/阶段1实验汇总表.md`

## 阶段2：真实 RWKV target + DSpark 双轨

### 实验

用纯 PyTorch 加载 RWKV-7 0.1B 权重作为 target，搭 DSpark 双轨架构（并行主干 cross-attn + RWKV-7 顺序头）。

### v1 → v2 → v3 优化路径

| 版本 | 优化点 | GRU 接受率 | RWKV 接受率 |
|---|---|---|---|
| v1 | cross-attn 单位置 hidden, 1500 步 | 0.316 | 0.327 |
| v2 | +8 位置 context, +置信度 BCE, 3000 步 | 0.535 | 0.701 |
| **v3** | +5 层 hidden, +5000 步, block 扫描 | - | **0.9795** |

### v3 block_size 扫描

| block | 平均 | 各位置 |
|---|---|---|
| 4 | 0.9795 | [0.934, 0.998, 0.998, 0.988] |
| 6 | 0.9652 | [0.840, 1.000, 0.998, 0.994, 0.994, 0.965] |
| 8 | 0.9734 | [0.805, 0.996, 1.000, 1.000, 0.996, 0.998, 0.996, 0.996] |

### 关键发现

1. **cross-attn context 长度是决定性优化**：单位置→8 位置让接受率从 33% 升到 70%
2. **RWKV-7 顺序头优势随优化放大**：v1 +0.011 → v2 +0.166（领先 31%）
3. **5 层 hidden + 5000 步达 97%+**：证明 RWKV 矩阵状态在累积前缀信息上远优于 GRU

详见：`参考/dspark_rwkv/阶段2实验报告.md`、`参考/dspark_rwkv/阶段2v2优化报告.md`

## 阶段4：端到端推理加速实测

### 阶段4a：batch=1（公式修正）

**发现并修正 stage4_e2e.py 的公式 bug**：原公式 `t_round = t_draft + (vl+1)*t_target` 错把验证当成 vl 次独立 forward。实际 target 一次 forward [B, T=vl] 就能验证 vl 个 token。

修正后 batch=1 加速比：1.58-3.84x（原错误报告 0.87x）。

### 阶段4b：并发实测（核心实验）

#### target forward 时间矩阵 (ms)

| batch\T | T=1 | T=4 |
|---|---|---|
| 1 | 49.14 | 70.62 |
| 8 | 31.60 | 42.25 |
| 32 | 35.86 | 39.27 |

#### 并发加速比（核心结果）

| verify_len | batch=1 | batch=2 | batch=8 | batch=32 |
|---|---|---|---|---|
| 1 | 1.70x | 1.67x | 1.56x | 1.61x |
| 2 | 2.04x | 2.04x | 2.12x | 2.39x |
| 3 | 2.66x | 2.89x | 2.40x | 3.16x |
| 4 | 2.99x | **3.75x** | 2.99x | 3.63x |

**最优配置**：batch=2, verify_len=4, 加速比 3.75x

### 加速来源分析

1. **target forward T=vl 时间增长缓慢**：T=1→T=4 只增 44%（非 4 倍），因为 GPU 并行处理多 token
2. **batch 增加摊薄 target 时间**：batch=1→8 下降 36%（GPU 利用率提升）
3. **draft 时间几乎不随 batch 变化**：6-8ms（draft 模型小，GPU 未饱和）

详见：`参考/dspark_rwkv/阶段3-4报告.md`

## DSpark 论文 vs 本实验

| 维度 | DSpark 论文 | 本实验 |
|---|---|---|
| target | Qwen3-4B | RWKV-7 0.1B |
| 顺序头 | GRU/RNN | RWKV-7 DPLR（领先 GRU 31%） |
| 接受率 | ~70-80% | 97%+（v3） |
| 加速比 | +60%~85% 用户速度 | 1.56-3.75x 吞吐 |
| 部署环境 | 多并发服务 | 单 GPU batch=N |

## 关键技术贡献

1. **验证 RWKV-7 DPLR 作为 DSpark 顺序头的可行性**：矩阵状态在累积前缀信息上远优于 GRU 向量状态
2. **token shift + LayerNorm 是必要条件**（消融实验证实）
3. **cross-attn context 长度是决定性优化**：单位置→8 位置让接受率翻倍
4. **5 层 hidden 采样 + 5000 步训练达 97%+ 接受率**
5. **修正推测解码加速比公式**：target 验证 vl 个 token 是 1 次 forward，不是 vl 次
6. **并发场景下达 3.75x 加速**：batch=2, verify_len=4

## 修正 memory 教训

memory 中原记录："单 GPU 推测解码无法加速"——基于阶段4a 的错误公式（0.87x）。

**修正后**：
- 单 GPU batch=1 下 DSpark 有 1.58-3.84x 加速（修正公式后）
- 并发 batch=N 下加速达 1.56-3.75x
- 原教训的错误根因：把 target 验证当成 vl 次独立 forward，高估了验证开销
